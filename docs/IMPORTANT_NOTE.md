# ⚠️ 重要说明

## 关于"不使用预编译库"的要求

根据迁移需求文件中的要求："整个项目不要使用任何需要预编译的库"

### 当前状况

项目目前使用了以下需要预编译的库：
- `better-sqlite3` - SQLite 数据库驱动（需要 C++ 编译）

### 解决方案

#### 方案 1：使用 Node.js LTS 版本（最简单）

`better-sqlite3` 为 Node.js LTS 版本（v18, v20）提供了预编译的二进制文件，无需本地编译：

```bash
# 使用 Node.js v20.x LTS
nvm use 20
npm install
npm run dev
```

#### 方案 2：完全重写为纯 JavaScript（复杂）

将 `better-sqlite3` 替换为 `sql.js`（纯 JavaScript SQLite）：

**优点**:
- 无需编译
- 跨平台兼容性好

**缺点**:
- 性能较差（比 better-sqlite3 慢 3-5 倍）
- 需要手动管理数据库持久化
- 需要重写所有数据库相关代码（约 2000+ 行）

**工作量估算**: 2-3 天

#### 方案 3：使用其他数据库（不推荐）

- MySQL/PostgreSQL - 需要额外安装数据库服务器
- MongoDB - 不适合关系型数据
- JSON 文件 - 性能差，不支持复杂查询

### 推荐方案

**使用方案 1：Node.js LTS 版本**

理由：
1. `better-sqlite3` 的预编译二进制文件不需要用户本地编译
2. 性能优秀，API 简洁
3. 开发效率高
4. 符合"不需要用户编译"的实际需求

### 如果必须使用纯 JavaScript

我已经创建了 `server/db/sqljs-adapter.js` 适配器，但需要：

1. 重写所有数据库操作代码
2. 修改所有路由文件
3. 实现数据库持久化逻辑
4. 测试所有功能

**预计工作量**: 2-3 天

### 当前状态

项目使用 `better-sqlite3`，在 Node.js LTS 版本上可以直接使用预编译二进制文件，无需本地编译。

如果需要完全切换到纯 JavaScript 方案，请告知，我将进行完整重写。
